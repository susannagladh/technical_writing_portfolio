> Note: FlowDesk Admin is a fictional product created for this portfolio sample.

## Overview

This sample shows examples of UX copy and microcopy for FlowDesk Admin, a fictional SaaS admin portal for managing users, roles and notification settings.

In this sample, **UX copy** means the text that appears in the product interface, such as button labels, field instructions, error messages, confirmation messages and empty states.

The copy helps administrators understand what is happening, make confident decisions and recover from common errors.

## Voice and tone

Write copy that is 

- clear
- concise
- helpful
- action-oriented
- calm when something goes wrong

The product should avoid technical jargon when a simpler explanation is available.

## Navigation and labels

Navigation labels and action labels should help administrators understand where they are and what each action does. The following examples use short, familiar terms that are easy to scan in an admin interface.

| UI element | Suggested UI text |
|---|---|
| Main navigation item | Users |
| Main navigation item | Roles |
| Main navigation item | Notifications |
| Main navigation item | Settings |
| Primary action button | Invite user |
| Secondary action button | Cancel |
| Destructive action button | Deactivate user |
| Confirmation action button | Save changes |

## Helper text

Helper text should give users enough context to complete a task without interrupting the workflow. The following examples explain what information to enter, what choice to make or what consequence an action has.

| Context | Suggested helper text |
|---|---|
| Role selection | Choose the access level this user needs. |
| Email field | Enter the email address the user will use to sign in. |
| Notification settings | Select which roles or groups should receive this notification. |
| Deactivate user | The user will lose access to FlowDesk, but their activity history will remain available. |

## Empty states

Empty states appear when there is no content to show. They should explain what happened and, when useful, suggest a next step.

### No users found

No users match your search.

Try another name or email address, or clear the filters to see all users.

### No pending invitations

There are no pending invitations.

When you invite a new user, their invitation status will appear here.

### No notifications enabled

No notifications are currently enabled.

Turn on notifications to keep users informed about account activity and access changes.

## Success messages

Success messages should confirm that an action was completed. They should be short, specific and placed close to the completed task.

| Scenario | Suggested message |
|---|---|
| User invited | Invitation sent to {example}. |
| Role updated | Role updated. |
| User deactivated | User deactivated. They can no longer sign in to FlowDesk. |
| User reactivated | User reactivated. They can sign in with their existing account. |
| Notification settings saved | Notification settings saved. |

## Error messages

Error messages should explain what went wrong and help the user recover. The following examples use plain language and include a clear next step when possible.

| Scenario | Suggested message | Recovery action |
|---|---|---|
| Invalid email address | Enter a valid email address. | Check the spelling and format of the email address. |
| User already exists | A user with this email address already exists. | Search for the user instead of sending a new invitation. |
| Invitation expired | This invitation link has expired. | Resend the invitation from the user details page. |
| Access denied | You do not have permission to view this page. | Contact an administrator if you need access. |
| Save failed | We could not save your changes. | Try again. If the problem continues, contact support. |

## Confirmation dialog

Confirmation dialogs should help users understand the consequence of an important or risky action before they continue.

### Deactivate user

**Title:** Deactivate user?

**Body:**  
This user will no longer be able to sign in to FlowDesk. Their historical activity will remain available in reports and audit logs.

**Primary action:** Deactivate user  
**Secondary action:** Cancel

## Before and after examples

The following examples show how unclear or generic interface text can be revised into more helpful UX copy.

### Example 1: Error message

**Before**

Something went wrong.

**After**

We could not save your changes. Try again. If the problem continues, contact support.

**Why this is better**

The revised message explains what failed and gives the user a clear next step.

### Example 2: Permission message

**Before**

Unauthorized.

**After**

You do not have permission to view this page. Contact an administrator if you need access.

**Why this is better**

The revised message uses plain language and explains how the user can resolve the issue.

### Example 3: Deactivation warning

**Before**

Are you sure?

**After**

This user will no longer be able to sign in to FlowDesk. Their historical activity will remain available.

**Why this is better**

The revised message explains the consequence of the action before the administrator confirms it.

## What this sample demonstrates

This sample demonstrates:

- UX writing for a technical product
- clear labels and action text
- helpful error messages
- user-focused recovery guidance
- confirmation copy for risky actions
- consistent terminology across product documentation
- plain-language communication for administrative workflows
